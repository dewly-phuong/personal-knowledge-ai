# Vectors Vec Index

## Description
The `vectors_vec` index is a critical component within the [[wiki/services/qmd|QMD (Query Markup Documents)]] data storage schema. It functions as an [[wiki/concepts/sqlite-vec|sqlite-vec]] vector index, specifically engineered to store and efficiently manage [[wiki/concepts/vector-embeddings|vector embeddings]] for semantic search capabilities.

## Purpose
The primary purpose of the `vectors_vec` index is to facilitate high-performance vector similarity searches. This capability forms one of the fundamental search backends within QMD's sophisticated hybrid search pipeline. It houses `content_vectors`, which are numerical representations capturing the semantic meaning of individual document chunks.

## Usage/Access
As an internal database component of [[wiki/services/qmd|QMD]], the `vectors_vec` index is not directly accessed by users or external systems. Instead, its functionality is exposed through QMD's command-line interface (CLI) and Software Development Kit (SDK):
*   **CLI**: Commands such as `qmd vsearch` (vector semantic search) and `qmd query` (hybrid search) leverage this index internally.
*   **SDK**: The `store.searchVector()` and `store.search()` methods within the [[wiki/services/qmd|QMD]] SDK interact with this index to perform vector-based retrieval.

## Technical Details
The `vectors_vec` index stores [[wiki/concepts/vector-embeddings|vector embeddings]] generated from chunks of indexed documents. Each entry in the index is identified by a `hash_seq` key, which links it back to specific `content_vectors` entries. This design enables QMD to perform rapid vector lookups and comparisons, crucial for identifying semantically similar documents or document segments within its knowledge base.

## Dependencies
*   [[wiki/concepts/sqlite-vec]]
*   [[wiki/services/qmd]] (as an integral part of its data storage and search capabilities)

## Role in QMD's Architecture
Within the [[wiki/services/qmd|QMD Hybrid Search Pipeline]], the `vectors_vec` index is actively queried during the "Vector Search" phase. It operates in parallel with [[wiki/concepts/bm25|BM25]] full-text search. The results from both search backends are then combined using Reciprocal Rank Fusion (RRF) and subsequently re-ranked by an LLM to deliver the highest quality and most relevant search results. The vector embeddings stored in this index are generated by a dedicated embedding model, such as `embeddinggemma-300M-Q8_0`, as part of QMD's indexing flow.
