# FTS Backend

## Technical Definition
A Full-Text Search (FTS) Backend is a component responsible for indexing and querying large volumes of text data to enable efficient keyword-based retrieval. In the context of [[wiki/services/QMD]], the FTS Backend specifically leverages the [[wiki/concepts/BM25]] algorithm.

## Architectural Design (within QMD)
Within the [[wiki/services/QMD]] architecture, the FTS Backend is one of the parallel retrieval mechanisms. It works alongside [[wiki/concepts/Vector Semantic Search]] to provide comprehensive search capabilities in a hybrid search pipeline.

### Indexing Flow
The FTS Backend's indexing process in [[wiki/services/QMD]] involves:
1.  **Collection & Glob Pattern**: Defining directories and file patterns (e.g., `**/*.md`) that constitute a document collection.
2.  **Markdown Files**: Processing the textual content of markdown files.
3.  **Title Parsing**: Extracting the main title from the document (typically the first heading or filename).
4.  **Content Hashing**: Generating a unique 6-character `docid` (document identifier) for each document.
5.  **Storage**: Storing the document content and associated metadata within a SQLite database.
6.  **FTS5 Indexing**: Creating a dedicated full-text search index using SQLite FTS5 for rapid keyword lookup based on the [[wiki/concepts/BM25]] algorithm.

### Query Flow
When a user initiates a `search` command in [[wiki/services/QMD]], the FTS Backend contributes to the overall result by:
1.  **Processing Query**: The user's query is passed to the FTS engine.
2.  **Retrieving Candidates**: Performing a keyword-based search against the `documents_fts` index, yielding a ranked list of documents based on their [[wiki/concepts/BM25]] scores.
3.  **Contributing to Fusion**: The results from the FTS backend are combined with results from the [[wiki/concepts/Vector Semantic Search]] using [[wiki/concepts/Reciprocal Rank Fusion (RRF)]] before an optional [[wiki/concepts/LLM Re-ranking]] step.

## Implementation Details
The FTS Backend in [[wiki/services/QMD]] is implemented using **SQLite FTS5**, which is integrated locally via `node-llama-cpp`.

### Score Normalization
Raw BM25 scores generated by SQLite FTS5 are positive values (ranging from 0 to approximately 25 or more). To facilitate fusion with other search backends (such as vector search), these raw scores are converted to a normalized range using `Math.abs(score)`.

## Consumers
The primary consumer of the FTS Backend is the [[wiki/services/QMD]] application itself, which exposes its search capabilities through various interfaces:
*   **CLI Commands**: Directly via `qmd search`.
*   **[[wiki/concepts/Model Context Protocol (MCP)]] Server**: For tighter integration with AI agents and other clients.
*   **Node.js/Bun SDK**: For programmatic usage within custom applications.
